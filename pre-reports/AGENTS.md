## Обязательные требования к качеству PRE-REPORT

Pre-report считается неготовым, если в нём есть формулировки вида:

- "требует проверки"
- "вероятно"
- "зависит от сценария"
- "по контексту коммита"
- "см. changelog"
- "детальные строки нужно сверить"
- "затронутые файлы и функции внутри них"
- "цель — исправить дефект или расширить поведение"
- любые другие обобщения без привязки к конкретному коду

Запрещено создавать pre-report, если:
- не прочитан полный diff коммита;
- не открыты изменённые файлы в `./projects`;
- не определены конкретные изменённые функции/методы/структуры/переменные;
- не описано, как меняется выполнение кода до и после коммита.

Если информации недостаточно, нужно не заполнять шаблон заглушками, а явно остановиться и написать:

`НЕ УДАЛОСЬ СОСТАВИТЬ PRE-REPORT: [конкретная причина]`

## Источники анализа

Для каждого утверждения нужно опираться на:

1. commit message;
2. полный diff из `./projects/changelogs`;
3. актуальный код проекта в `./projects`;
4. соседние функции/модули, если они участвуют во взаимодействии.

Нельзя заменять анализ кода пересказом commit message.

## Что считать изменённой сущностью

Нельзя создавать сущность типа:

- "коммит";
- "набор изменений";
- "затронутые файлы";
- "логика коммита";
- "подсистема".

Сущностью может быть только конкретный элемент кода или контракта:

- функция;
- метод;
- класс;
- структура;
- интерфейс;
- поле структуры;
- переменная;
- константа;
- конфигурационный параметр;
- CLI-аргумент;
- API endpoint;
- протокол;
- формат данных;
- ветка выполнения;
- обработчик ошибки;
- тестовый кейс.

Если изменён только один `if`, сущностью является не "коммит", а конкретная функция и конкретная ветка условия внутри неё.

## Требования к полю Diff

В поле `Diff` нельзя писать:

`См. changelog`

Нужно вставлять релевантный фрагмент diff полностью или сокращённо, но так, чтобы были видны конкретные изменённые строки.

- старый код
+ новый код
