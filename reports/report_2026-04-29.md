# Обзор обновлений за 29 апреля 2026

Ниже собраны изменения из `projects/changelogs` за **2026-04-29** с фокусом на функциональность, важную для эксплуатации прокси-инфраструктуры, обхода блокировок и устойчивости к деградации под нагрузкой.

## telemt: расширение TLS-доменов, усиление валидации и стабилизация hot-path

В `telemt` в этот день прошла серия изменений вокруг TLS-fronting, выдачи клиентских ссылок и внутренних ограничителей. Это важный слой, потому что он одновременно влияет на:

- маскировку трафика (через выбор SNI/домена);
- корректность клиентского онбординга (какие ссылки получает пользователь);
- устойчивость ноды под высокой конкуренцией (кэш, лимиты, таймауты).

Ранее API в блоке `users` отдавал классические/secure/tls-ссылки, но не давал структурированного соответствия «какой именно TLS-домен → какая ссылка». Для панелей и автоматизации это создавало лишнюю неоднозначность: домен есть в конфиге, ссылка есть в массиве, но явного соответствия не было.

Теперь в модель добавлен список `tls_domains` как массив пар `{domain, link}`. На уровне генерации ссылок добавлена отдельная логика выбора «дополнительных» TLS-доменов: из `censorship.tls_domains` исключаются пустые значения, primary-домен и дубликаты, после чего для каждого валидного домена формируется отдельная tg-ссылка. В результате внешние системы получают не просто набор ссылок, а прямо нормализованный mapping, пригодный для UI и интеграций.

Следующий важный блок — bootstrap TLS fetcher для мультидоменных конфигураций. В старой схеме при определённых настройках `mask_host` все домены могли проходить через один и тот же host-путь, что ухудшало предсказуемость и полноту прогрева кэша для `tls_domains`. Изменение вводит отдельную функцию выбора host для каждого домена: если `mask_host` совпадает с primary TLS-доменом, то для fetch используется текущий домен итерации; если `mask_host` задан явно и не равен primary, то он сохраняется как есть. Это даёт более корректный прогрев и снижает риск перекоса, когда часть доменов фактически не проходит «собственным» путём.

Параллельно в кэше TLS-front введена проверка соответствия сертификатных метаданных домену (`SAN`/`CN` + wildcard-логика). Если запись кэша содержит сертификат, не подходящий к домену, запись пропускается. Практически это уменьшает шанс использовать «формально успешный», но семантически неверный кэш, что особенно критично в средах с активным DPI и нестабильными upstream-путями.

Отдельно улучшена бухгалтерия лимита «полного сертификата на IP»: добавлен редкий sweep протухших записей вместо чистки на каждом вызове. Это снижает издержки на горячем пути и при этом сохраняет TTL-политику. Также видны тесты на инварианты sweep-а (когда должен/не должен запускаться), что снижает риск регрессий в логике ограничителя.

Наконец, в блоке timeout-инвариантов tightened-валидация: `general.me_route_blocking_send_timeout_ms` теперь должен быть в диапазоне `[1, 5000]` (ноль запрещён). Это устраняет «пограничное» состояние с нулевым таймаутом, которое могло вести к некорректному поведению отправки в блокирующем маршруте.

**Итог для эксплуатации:**

- администраторы получают более прозрачную работу с множеством TLS-доменов и более полезный API-вывод для автоматизации;
- повышается консистентность TLS-front кэша и снижается риск ошибок из-за mismatch сертификата;
- hot-path становится экономнее за счёт контролируемого sweep;
- конфигурационные ошибки с нулевым timeout теперь отсекаются заранее.

## TrustTunnel / TrustTunnelClient: правки CI-пайплайнов benchmark и релизные инкременты

В `TrustTunnel` и `TrustTunnelClient` изменения за 29 апреля в основном инфраструктурные: корректировка прав в GitHub Actions для jobs, которые комментируют PR/реагируют на события, и стандартные version bump от Bamboo.

Контекст: benchmark/workflow-цепочки часто взаимодействуют с PR-объектами (комментарии, реакции, отчёты). Без явно выставленных прав `pull-requests: write` такие действия могут ломаться или становиться зависимыми от внешних токенов/секретов. В observed changes добавлены права для соответствующих job-ов и местами убрано использование отдельного токена в пользу штатного механизма прав workflow.

Для runtime прокси-логики эти изменения напрямую не вводят новый транспортный функционал, но они важны операционно: стабильная публикация benchmark-результатов и корректная автоматизация ревью/релизного контура уменьшают вероятность «тихих» сбоев в инженерном процессе.

Version bump-коммиты (`skipci: Automatic version increment by Bamboo`) затрагивают упаковку и метаданные релизов (Android/iOS/Windows/Conan и т.д.) и фиксируют переход на очередные версии артефактов.

## Что важно проверить команде после этих изменений

1. Для telemt:
   - что внешние панели/боты, потребляющие `users` API, начали использовать `tls_domains` как первичный источник соответствия домен→ссылка;
   - что в прод-конфигах список `censorship.tls_domains` не содержит «мусорных» значений (пустые/дубли), хотя теперь часть из них фильтруется;
   - что `me_route_blocking_send_timeout_ms=0` не используется в старых шаблонах конфигов.

2. Для TrustTunnel и TrustTunnelClient:
   - что benchmark workflow действительно оставляет комментарии/реакции в PR без ручного токена;
   - что цепочка релизных метаданных (версии в podspec/gradle/rc/conandata) остаётся синхронной после автоматических bump.

Если нужно, следующий шаг — сделать точечный отчёт только по telemt с трассировкой стеков функций (API users → генерация links → TLS bootstrap → кэш) и сценариями «до/после» на конкретных конфигурациях.
