# Отчёт по обновлениям за 27 апреля 2026

27 апреля 2026 года в changelog-ах зафиксированы изменения в трёх проектах: `TrustTunnel`, `TrustTunnelClient` и `telemt`. Ключевые технически значимые изменения относятся к `TrustTunnel` (исправление поведения QUIC/H3 на Linux) и `TrustTunnelClient` (перенос имени сервера в сам конфиг и корректировка API тестового приложения). В `telemt` за эту дату зафиксировано только редактирование README.

## TrustTunnel: исправление H3-тестов на Linux и корректная обработка QUIC timeout

В контексте `TrustTunnel` речь идёт о тестовом контуре HTTP/3 (H3) поверх QUIC, который проверяет reverse proxy и ping-сценарии. Здесь критично, чтобы механизм восстановления потерь (loss recovery) срабатывал не только при приходе новых UDP-пакетов, но и при истечении таймера дедлайна QUIC-соединения. Иначе при потере пакетов большие H3-передачи могут зависать до idle timeout сервера.

Внутренняя механика до исправления была такой: код ждал `socket.readable()` с таймаутом `quic_conn.timeout()`, но в ряде участков таймаут просто игнорировался (результат `timeout(...).await` не проверялся), поэтому `quic_conn.on_timeout()` не вызывался. Это видно в изменениях `lib/tests/common/mod.rs`, где одинаковый паттерн исправлен во множестве мест: теперь при `is_err()` после `tokio::time::timeout(...)` явно вызывается `self.quic_conn.on_timeout()`. Одновременно в `lib/src/quic_multiplexer.rs` добавили новое сообщение `SocketMessage::RefreshDeadline`, которое после flush принудительно обновляет дедлайн из `quiche::Connection::timeout()`. Это важно, потому что flush может менять состояние recovery/таймеров, и мультиплексор должен пересчитать ближайший дедлайн, а не жить со старым значением.

Отдельно переработана отправка QUIC-датаграмм: функция `flush_pending_data(...)` теперь отправляет пакет на `send_info.to`, полученный из `quic_conn.send(...)`, вместо старого фиксированного peer-адреса. Это снижает риск ошибок при маршрутизации/миграции адреса в рамках QUIC-сессии и делает тестовое поведение ближе к реальной модели QUIC.

Изменение в тестовой оркестрации `lib/tests/reverse_proxy.rs` убирает прежние ограничения `#[cfg(target_os = "macos")]` для H3-тестов и фиксирует lifetime задач: `endpoint_task` теперь пинится и удерживается в `select!`, чтобы endpoint не завершился раньше времени и не оборвал in-flight H3-потоки. Ранее в таких гонках тест мог падать как будто из-за сети, хотя первопричина была в управлении задачами и timeout-обработке.

Практический эффект:
- пользователи и CI получают более стабильные H3-сценарии на Linux;
- разработчикам нужно учитывать, что дедлайн QUIC теперь явно обновляется после flush и timeout-событий;
- администраторам VPN-инфраструктуры это снижает риск ложных выводов о «нестабильном QUIC» при тестировании прокси/туннеля.

В тот же день также зафиксирован отдельный `skipci`-коммит с автоинкрементом версии endpoint-пакета (`1.0.35 -> 1.0.36`) без функциональной логики.

## TrustTunnelClient: имя сервера берётся из TOML-конфига, а не из отдельного аргумента API

Этот блок изменений относится к клиентским адаптерам (Apple/Android/Flutter testapp), где приложение передаёт конфиг в нативный слой для запуска VPN. До обновления имя сервера прокидывалось отдельным параметром метода `start(...)`, параллельно с полным TOML-конфигом. Это дублировало источник данных: часть endpoint-метаданных жила в конфиге, а часть — в отдельном аргументе вызова.

Под капотом теперь сделана унификация источника истины:
- в Swift-модели `Endpoint` добавлено поле `name`, а `dns_upstreams` сделано опциональным (`[String]?`), чтобы корректно обрабатывать конфиги без этого ключа;
- `VpnManager.updateConfiguration(...)` и `start(...)` больше не принимают `serverName`, а парсят `VpnConfig` и заполняют `configuration.serverAddress` из `vpnConfig.endpoint.name`;
- во Flutter/Pigeon API сигнатуры `NativeVpnInterface.start(...)` в generated-файлах и реализациях Android/Swift/Windows сведены к единственному аргументу `config`.

Это устраняет класс ошибок рассинхронизации, когда UI передал одно имя сервера аргументом, а в TOML указано другое. После изменения поведение детерминировано: отображаемое/используемое имя сервера всегда берётся из `[endpoint].name` внутри того же конфига, который реально запускает туннель.

Дополнительно поправлено поведение DNS-настроек в Apple-адаптере: выражение для выбора DNS теперь корректно работает и при отсутствии `dns_upstreams` (через `?? []`). То есть отсутствие поля трактуется как «список пуст», и используется дефолтный набор AdGuard DNS вместо ошибки декодирования/падения логики.

Практический эффект:
- пользователи тестового приложения меньше сталкиваются с расхождением между UI-параметрами и реальным подключением;
- разработчикам нужно учитывать breaking-like изменение сигнатуры вызова `start` в межплатформенном API (остался только `config`);
- администраторам VPN-сервисов важно, что server display name теперь должен быть заполнен в `[endpoint].name` в передаваемом конфиге.

В этот же день есть отдельный `skipci`-коммит автоинкремента версии (`1.0.54 -> 1.0.55`) в сборочных манифестах без новой прикладной логики.

## Что важно по итогам дня

Главный эксплуатационный итог 27 апреля 2026 — исправление timeout/retransmission-поведения для QUIC/H3 в `TrustTunnel` и снятие платформенного ограничения H3-тестов для Linux. Второй по значимости блок — нормализация конфигурационного контракта в `TrustTunnelClient`, где источник server name теперь единый и находится в TOML-конфиге (`[endpoint].name`).
