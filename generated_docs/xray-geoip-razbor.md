# Что такое GeoIP в Xray: устройство и варианты использования

## Коротко

**GeoIP в Xray** — это механизм маршрутизации по IP-диапазонам, сгруппированным по коду (обычно по стране, например `CN`, `US`) в бинарном файле `geoip.dat`.

Когда в правиле роутинга вы пишете `geoip:cn`, Xray:

1. Преобразует эту запись во внутренний формат `ext:geoip.dat:CN`.
2. Находит в `geoip.dat` набор CIDR-диапазонов для кода `CN`.
3. Строит IP-матчер.
4. При обработке соединения сравнивает IP назначения (или source/local IP — в зависимости от поля правила) с этими диапазонами.

---

## Где это видно в коде

### 1) Разбор строки правила (`geoip:...`, `ext:...`, CIDR)

Разбор делается в `common/geodata/rule_parser.go`:

- `geoip:xx` автоматически конвертируется в `ext:geoip.dat:xx`.
- Поддерживаются префиксы `ext:` и `ext-ip:` для явного указания файла и кода.
- Если это не `geoip/ext`, строка трактуется как обычный CIDR (например `192.168.0.0/16` или `fe80::/64`).
- Префикс `!` инвертирует условие (reverse match), причём логика корректно учитывает и `!` перед правилом, и `!` перед кодом.

Практически это значит:

- `geoip:cn` — матч по диапазонам `CN` из `geoip.dat`;
- `!geoip:cn` — матч «всё, что не CN»;
- `ext:mygeo.dat:corp` — матч по кастомному файлу `mygeo.dat`, код `CORP`;
- `10.0.0.0/8` — матч по конкретному CIDR без GeoIP-базы.

### 2) Как читается `geoip.dat`

Загрузка выполняется в `common/geodata/geodat_loader.go`:

- файл открывается через asset-слой (`filesystem.OpenAsset`),
- ищется нужный `code` внутри geodata-файла,
- найденный protobuf-блок десериализуется в `GeoIP`,
- из него берутся CIDR (`geoip.Cidr`).

Там же есть проверка, что код реально существует в указанном файле (`checkFile`).

### 3) Откуда берётся путь к файлу `geoip.dat`

Asset-поиск реализован в `common/platform/others.go` и `common/platform/filesystem/file.go`:

- сначала используется директория из переменной окружения `xray.location.asset`;
- затем проверяются стандартные пути (`/usr/local/share/xray/`, `/usr/share/xray/`, `/opt/share/xray/`).

Если вы используете `ext:geoip.dat:cn`, Xray ищет именно этот файл в asset-локациях.

### 4) Где GeoIP участвует в маршрутизации

В `infra/conf/router.go` поле JSON `ip` (а также `sourceIP`, `localIP`) разбирается через `geodata.ParseIPRules`.

Далее в `app/router/config.go` создаются IP-условия:

- `ip` → матчер по **целевым** IP (`MatcherAsType_Target`),
- `sourceIP` → по **исходным** IP (`MatcherAsType_Source`),
- `localIP` → по **локальным** IP (`MatcherAsType_Local`).

Сами условия применяются в `app/router/condition.go` через `AnyMatch(...)`.

Важно: в том же файле есть предупреждение, что `localIP` для UDP может работать ограниченно при inbound на всех интерфейсах.

### 5) Кэш и перезагрузка GeoIP-матчеров

`common/geodata/ip_registry.go` хранит динамические IP-матчеры и умеет пересобирать их через `Reload()`.

Это используется приложением `app/geodata` (`app/geodata/geodata.go`):

- можно настроить cron-перезагрузку,
- при необходимости скачать обновлённые geodata assets,
- затем выполнить reload matchers без полного рестарта логики сопоставления правил.

Конфигурация этого механизма описана в `app/geodata/config.proto`:

- `cron` — расписание,
- `outbound` — через какой outbound скачивать,
- `assets` — список `url/file` для geodata-файлов.

## Форматы правил GeoIP, которые поддерживает Xray

Ниже — форматы, подтверждённые парсером и тестами (`common/geodata/rule_parser_test.go`):

1. **Короткий формат**
   - `geoip:cn`
   - `geoip:us`

2. **Явный формат с файлом**
   - `ext:geoip.dat:cn`
   - `ext-ip:geoip.dat:cn`

3. **Инверсия**
   - `!geoip:cn`
   - `geoip:!cn`
   - `!!geoip:cn` (двойная инверсия даёт исходный смысл)

4. **Без GeoIP (CIDR как альтернатива)**
   - `192.168.0.0/24`
   - `fe80::/64`
   - `192.168.0.1` (хост-маска по умолчанию)

## Типовые варианты использования

## 1) Разделение трафика по странам

Идея: локальный регион пускать напрямую, остальное — через прокси/туннель.

Пример:

```json
{
  "type": "field",
  "ip": ["geoip:cn"],
  "outboundTag": "direct"
}
```

Что делает: если целевой IP попадает в `CN` из `geoip.dat`, выбирается `direct`.

---

## 2) Обратная логика (всё кроме страны)

```json
{
  "type": "field",
  "ip": ["!geoip:cn"],
  "outboundTag": "proxy"
}
```

Что делает: всё, что не `CN`, отправляется в `proxy`.

---

## 3) Контроль по sourceIP/localIP (серверные и сложные сценарии)

GeoIP можно применять не только к target IP:

- `sourceIP` — фильтрация по IP клиента;
- `localIP` — фильтрация по локальному IP интерфейса.

Это полезно для multi-homed серверов и специфичных политик, но для `localIP` + UDP есть ограничение (см. предупреждение в коде).

---

## 4) Кастомная geodata-база

Если стандартного `geoip.dat` мало, можно использовать свой файл и код:

```json
{
  "type": "field",
  "ip": ["ext:mygeo.dat:corp"],
  "outboundTag": "corp-egress"
}
```

Требование: `mygeo.dat` должен быть доступен в asset-пути Xray.

---

## 5) Без GeoIP вообще: явные CIDR-правила

Если нужна полностью контролируемая и предсказуемая политика, можно обойтись CIDR:

```json
{
  "type": "field",
  "ip": ["203.0.113.0/24", "2001:db8::/32"],
  "outboundTag": "direct"
}
```

Плюс: не зависите от актуальности внешней geodata-базы.

---

## Практические нюансы

1. **GeoIP работает по IP, не по доменному имени.**  
   Если у вас rule по `ip`, то сначала должен быть известен IP назначения (через DNS-резолв и стратегию маршрутизации).

2. **Коды нормализуются в верхний регистр.**  
   `geoip:cn` и `geoip:CN` эквивалентны.

3. **Ошибка в коде/файле выявляется на этапе парсинга/загрузки.**  
   Xray проверяет, что нужный код реально существует в файле geodata.

4. **Обновления geodata можно автоматизировать.**  
   Через `app/geodata` можно периодически обновлять файлы и reload-ить матчеры.

5. **Производительность учитывается в реализации матчеров.**  
   В `ip_matcher.go` есть оптимизации по структурам IP-set и эвристикам, чтобы не сканировать всё «в лоб».

---

## Когда выбирать GeoIP, а когда CIDR

Используйте **GeoIP**, если:

- нужна быстрая политика уровня «по странам/категориям»;
- хотите меньше ручного сопровождения правил;
- готовы поддерживать актуальность geodata-файлов.

Используйте **CIDR**, если:

- нужна детерминированная и полностью контролируемая маршрутизация;
- есть свой список сетей/ASN/сегментов;
- важно избежать зависимости от внешних geodata-наборов.

На практике часто применяют гибрид: базовый слой на GeoIP + точечные исключения CIDR-правилами.

---

## Итог

GeoIP в Xray — это не «внешний API», а встроенный механизм сопоставления IP с предзагруженными диапазонами из geodata-файлов (`geoip.dat`), который интегрирован в маршрутизатор правил Xray и может работать для target/source/local IP. Для продакшна ключевое — следить за актуальностью geodata и явно понимать, где GeoIP удобнее, а где надёжнее CIDR.
