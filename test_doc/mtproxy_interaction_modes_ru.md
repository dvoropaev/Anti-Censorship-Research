# MTProxy: как он взаимодействует с клиентом и с серверами Telegram

> Режим: **DOC + TEST**.  
> Этот материал основан на исходниках и документации в репозитории `projects/telegram/MTProxy`.

## 1) Что такое MTProxy в этой архитектуре

`MTProxy` — это официальный MTProto proxy от Telegram (C-код), который:

1. Принимает клиентские подключения Telegram по TCP.
2. После проверки клиентского секрета передаёт пользовательский трафик в инфраструктуру Telegram (через middle proxy endpoints из конфига).
3. Отдельно поднимает локальный HTTP endpoint статистики на loopback-порту (`-p`, пример `/stats`).

## 2) Как MTProxy взаимодействует с клиентом

### 2.1 Вход и классификация клиента

На входе сервер принимает TCP-сессию от Telegram-клиента и сопоставляет её с допустимыми proxy-secret (параметры `-S`).

Логика по сути делится на 2 ветки:

- **Валидный MTProxy клиент** → сессия принимается и трафик маршрутизируется дальше в Telegram.
- **Невалидный клиент/секрет** → сессия отвергается.

### 2.2 Какие клиентские режимы поддерживаются

В доступных источниках этого репозитория явно подтверждаются:

- стандартный режим MTProxy с user-secret (`-S ...`);
- режим **random padding** для клиентов, которые запрашивают его через префикс `dd` в секрете.

### 2.3 Что происходит в режиме random padding (`dd`)

`dd`-префикс в клиентском секрете включает добавление случайного паддинга к пакетам. В README это прямо описано как защита от детекта по размерам пакетов у части провайдеров/DPI.

Важно: это не отдельный транспортный протокол, а модификация поведения MTProxy-трафика (добавление случайного заполнения).

### 2.4 Что происходит после handshake клиента

После валидации клиентского секрета сервер работает как relay:

- принимает MTProto-пакеты от клиента;
- инкапсулирует/передаёт их на выбранные upstream-цели Telegram;
- поддерживает обратный поток ответов клиенту.

## 3) Как MTProxy взаимодействует с серверами Telegram

Нужно разделять **два класса взаимодействий**.

### 3.1 Контрольные служебные запросы (Control Plane)

Оператор MTProxy получает у Telegram служебные данные:

- `getProxySecret` (секрет для связи прокси с инфраструктурой Telegram);
- `getProxyConfig` (актуальная конфигурация upstream-узлов, например `proxy_for ...`).

По README это делается внешне (обычно `curl`), затем файлы передаются процессу через `--aes-pwd proxy-secret proxy-multi.conf`.

### 3.2 Data Plane: транспорт пользовательского трафика

Рабочий трафик клиентов уходит с MTProxy на Telegram upstream-узлы, которые загружены из proxy-конфига (строки вида `proxy_for dc ...`).

В исходниках видно:

- парсинг и разбор конфигурации в `mtproto-config.c`;
- построение/обновление кластеров целей по DC;
- рабочая пересылка пакетов через forward-функции в `mtproto-proxy.c`.

## 4) Как MTProxy получает адреса серверов Telegram

### 4.1 Адреса для Middle Proxy (upstream targets)

Источник адресов: файл `proxy-multi.conf`, который обычно берут через `getProxyConfig`.

- Парсер обрабатывает `proxy_for <dc> <ip>:<port>;`.
- Поддерживается `default`-кластер.
- Цели группируются по DC-кластерам.

### 4.2 Адреса Telegram DC для direct-маршрута

В данном репозитории MTProxy основная модель — работа по конфигурации upstream proxy targets из `proxy-multi.conf`.

Явной отдельной пользовательской модели «direct route в DC» (как отдельного режимного переключателя) в просмотренных источниках не зафиксировано.

### 4.3 DNS/резолвинг

При разборе server:port в конфиге используется резолвинг hostname (через `gethostbyname` в коде парсинга конфигурации), то есть возможны как IP, так и доменные имена в конфиге.

## 5) Какие протоколы используются

Ниже разложено по каналам взаимодействия.

### 5.1 Клиент ↔ MTProxy

- Транспорт: **TCP**.
- Протокол: **MTProto proxy transport** (MTProxy).
- Опционально: random padding для клиентов с `dd`-секретом.

### 5.2 MTProxy ↔ Telegram (служебные загрузки)

- В типовом сценарии из README: служебные данные скачиваются оператором по **HTTPS** (`curl` к `core.telegram.org`) до запуска процесса.
- Сам процесс затем читает уже сохранённые локальные файлы (`proxy-secret`, `proxy-multi.conf`).

### 5.3 MTProxy ↔ Telegram (передача пользовательского трафика)

- Транспорт: TCP-соединения к upstream-targets из `proxy-multi.conf`.
- Логика: проксирование MTProto-трафика между клиентом и Telegram-инфраструктурой.

> Важно: в рамках этого репозитория надёжно подтверждается архитектура через proxy-config clusters и relay. Детализация wire-форматов Telegram-протокола в документации репозитория ограничена.

## 6) Какие есть режимы для каждого вида взаимодействия

Ниже — практическая матрица.

## 6.1 Для клиентского входа (Client → MTProxy)

1. **Базовый MTProxy-режим** (секреты `-S`).
2. **MTProxy + random padding** (клиентский секрет с префиксом `dd`).

Эти режимы управляются форматом секрета и параметрами запуска.

## 6.2 Для реакции на невалидный/чужой трафик

- Если handshake/secret не проходит валидацию — соединение отвергается.
- В просмотренных источниках официального MTProxy нет отдельного режима маскировки не-MTProxy трафика наподобие `mask_host/mask_port` из Telemt.

## 6.3 Для маршрута к Telegram (MTProxy → Telegram Data Plane)

1. **Маршрут через upstream targets из `proxy-multi.conf`** (основной рабочий путь).

## 6.4 Для доступности маршрута при проблемах upstream

- Используются внутренние механики кластеров/целей и перезагрузка конфигурации.
- Отдельного флага вида `me2dc_fallback` (как в Telemt) в просмотренных источниках MTProxy не подтверждено.

## 6.5 Для orchestration и эксплуатации

Практически важные режимы/параметры запуска:

- `-M` — число worker-процессов;
- `-p` — локальный порт статистики (loopback endpoint);
- `-H` — публичный порт для клиентов;
- `-S` — один или несколько клиентских секретов;
- `-P` — proxy tag (после регистрации у `@MTProxybot`).

## 7) Ответы на ваши вопросы в коротком виде

### Как MTProxy взаимодействует с клиентом?

Через TCP; клиент предъявляет MTProxy-secret, после валидации сервер запускает relay пользовательского MTProto-трафика.

### Как MTProxy взаимодействует с серверами Telegram?

- Служебно: использует полученные от Telegram `proxy-secret` и `proxy-config` (обычно заранее скачанные оператором).
- Для трафика: отправляет клиентские данные в upstream-кластеры Telegram из `proxy-multi.conf`.

### Как получает адреса серверов Telegram?

- Из `proxy-multi.conf` (`proxy_for ...`), который берётся у `getProxyConfig`.
- Hostname-адреса при необходимости резолвятся в парсере.

### Какие протоколы используются?

- Клиентский вход: MTProxy/MTProto proxy over TCP.
- Служебные загрузки конфигов: HTTPS (внешний шаг подготовки).
- Апстрим трафика: TCP к upstream-targets Telegram.

### Какие есть режимы для каждого вида взаимодействия?

- Клиентские: базовый MTProxy и вариант с random padding (`dd`).
- Поведение на невалидный трафик: reject.
- Маршрут к Telegram: через proxy-config targets.
- Эксплуатационные режимы: worker count, stats port, набор секретов, proxy tag.

---

## 8) Ограничения и что важно не перепутать

1. **User secret** (`-S ...`) и **proxy-secret** (`getProxySecret`, `--aes-pwd`) — разные сущности для разных уровней.
2. `dd` — это режим random padding, а не отдельный «новый протокол».
3. `proxy-multi.conf` — это критичный источник истины для upstream-маршрутизации MTProxy.
4. В этом репозитории есть код парсинга/маршрутизации и relay, но не полная внешняя спецификация всех внутренних wire-деталей Telegram transport.
