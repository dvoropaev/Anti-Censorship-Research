# Обзор изменений в upstream-проектах за 2026 год

> Период анализа: коммиты из `./upstreams/changelogs/*.log` с датами `2026-*`, плюс сверка по исходному коду в `./upstreams/*`.

## 1) Краткий обзор: главные тренды 2026

2026 год в этих проектах прошёл под знаком **практического усиления устойчивости к блокировкам**, и это видно сразу по нескольким линиям:

1. **Сдвиг от “просто работает” к “труднее детектировать и ломать”**:
   - В `Xray` выросло внимание к TLS/ECH/пиннингу и к новым транспортам (Hysteria2 + Salamander + TUN).
   - В `telemt` резко усилены маскировка и anti-probe логика (много security/adversarial тестов и ужесточение FakeTLS-пайплайна).
   - В `dpibreak` и `zapret` — практические правки обходных механизмов (QUIC распознавание, fragmentation/TTL, стабильность низкоуровневых хуков).

2. **Больше “операторских” изменений для продакшена VPN-сервисов**:
   - Connection limits, watchdog/idle timeouts, детальная диагностика, API-методы управления правилами, hot-reload, стабильный CI/release.
   - В TrustTunnel-ветке: глубокая доработка deep-link и конфиг-цепочки (server name / dns_upstreams / client_random_prefix / custom_sni), плюс security fixes против утечек в LAN.

3. **API/схемы конфигов активно эволюционируют**:
   - `Xray`: депрекации и переходы (например, уход от allowInsecure в пользу pinning/verify).
   - `sing-box`: удаление просроченных deprecated-фич, расширение правил/действий (bypass, pre-match, авто-маршрутизация).
   - `TrustTunnelClient/Flutter`: миграции полей deep-link (`server_display_name -> name`, `dns_servers -> dns_upstreams`).

4. **Безопасность стала “процессом”, а не отдельным патчем**:
   - Наблюдается не просто “fixed bug”, а системный подход: тестовые матрицы, adversarial сценарии, hardening, более строгая валидация входов, усиление изоляции маршрутов и local-socket защиты.

## 2.1 Ядра и транспорты обхода: Xray + sing-box

## Xray-core: самые важные изменения 2026

### Что добавили

#### 1) Полноценный TUN inbound (Windows/Linux/Android + затем macOS)
**Что это даёт:**
- Можно строить схему “системный трафик → TUN → Xray policy/routing”, то есть Xray становится системным dataplane, а не только прокси на порту.
- Для VPN-сервисов это упрощает унификацию политики (split/full tunnel, DNS, QUIC/UDP, метки).

**Как это устроено внутри:**
- В конфиг-пайплайне inbound `tun` обрабатывается отдельно: для него не требуется классическая `port`-привязка (потому что вход — сетевой интерфейс, не TCP-listener).
- В обработчике TUN трафик инкапсулируется в routing session и диспетчеризуется как обычный connection flow.

#### 2) Hysteria v2 outbound + udphop + Salamander mask
**Что это даёт:**
- Более “живучий” транспорт в средах с агрессивной фильтрацией UDP/QUIC.
- Salamander/udpmask добавляет слой маскировки UDP-профиля.

**Для администраторов:**
- Это сильный вариант для fallback-пула: когда обычный TLS/VLESS-path начинает блокироваться по сигнатурам.
- Но важно корректно выставлять версию и параметры congestion/interval, иначе можно получить нестабильность.

#### 3) API расширение: ListRule()
**Что это даёт:**
- Автоматизация управления маршрутизацией через внешний контроллер (панели/оркестратор).
- Удобнее делать drift-control между “желательной” и “фактической” политикой маршрутов.

### Усиление маскировки и обхода блокировок

#### 4) ECH/TLS улучшения
- Изменения по ECH (включая корректировку поведения `echForceQuery`) и обновления uTLS/reality/quic-go повышают шансы пройти DPI, который комбинирует TLS fingerprint + ALPN + поведенческие признаки.

#### 5) Переход от allowInsecure к pinning/verify
**Ключевой момент:**
- В коде есть явная логика “после 2026-06-01 `allowInsecure` считается removed feature”.
- Рекомендованный путь: `pinnedPeerCertSha256` и `verifyPeerCertByName`.

**Почему это важно:**
- Многие инсталляции historically жили на `allowInsecure=true` для “быстрого старта”.
- Теперь это операционный риск: надо заранее мигрировать конфиги, иначе при обновлении можно получить массовые падения соединений.

### Что починили (баги/безопасность)

- Исправления утечек UDP в SS2022 outbound.
- Серия фиксов TUN (контексты, FullCone NAT, sniffing/ports/logging, stall recovery).
- Исправления в DNS/router (включая кейсы panic и ошибки pack/parse).

### На что обратить внимание VPN-администратору

1. Проверить все профили на `allowInsecure` и подготовить миграцию до hard-removal.
2. Если используете Hysteria2, валидировать настройки congestion/udphop на реальных каналах с packet loss.
3. Для TUN-сценариев прогнать тесты: DNS-leak, split routing, QUIC fallback, long-lived TCP.
4. Использовать API ListRule для периодической верификации policy drift.

## sing-box: самые важные изменения 2026

### Что добавили / расширили

#### 1) Крупный функциональный пакет 1.13.x
По changelog и коду видно усиление в нескольких слоях:
- NaiveProxy outbound,
- pre-match + action `bypass` для `auto_redirect`,
- новые rule items (`preferred_by`, interface-address семейство, `wifi_ssid`/`wifi_bssid`),
- Tailscale endpoint: `advertise_tags`, relay options, system interface,
- TLS опции (в т.ч. kTLS, pinned key/ECH-query поле),
- `bind_address_no_port` для масштабируемых исходящих dial-сценариев.

**Что это даёт practically:**
- Гораздо более тонкую политику маршрутизации с привязкой к состоянию интерфейса/сети.
- Улучшенную маскировку и устойчивость (Naive/ECH-path, расширенные TLS/route-инструменты).
- Больше “рычагов” для high-concurrency нод (особенно Linux edge).

#### 2) OCM/CCM сервисный слой
- Добавлены/расширены Multiplexer-сервисы (включая WebSocket для Responses API в OCM).
- Это уже не только “прокси-ядро”, но и инфраструктурная прослойка для удалённого доступа к LLM tooling.

### Усиление маскировки/обхода

- Ставка на NaiveProxy и отказ от уTLS как универсальной “панацеи” (прямо отражено в changelog-объяснениях).
- Улучшения `auto_redirect` и bypass-предматча: можно точнее исключать/перенаправлять потоки до тяжёлой обработки.

### Важные фиксы

- DNS (deadlock/recursion/cname chain/legacy defaults/fake-ip),
- Tailscale system interface и связанный routing,
- утечки goroutine/websocket в Clash API,
- netns fd leak и ряд platform-specific regression fix.

### Изменения API/конфиг-поведения

- Удалены просроченные deprecated-фичи.
- Часть полей/семантики теперь строже валидируется (что хорошо для безопасности, но требует аккуратной миграции).

### Что важно администратору

1. Пересобрать “базовые шаблоны конфигов” под 1.13+ (особенно route/pre-match/auto_redirect).
2. Проверить совместимость старых UI/генераторов конфигов с удалёнными deprecated полями.
3. Для Naive/ECH путей мониторить размер TLS fingerprint diversity и фактическую блокируемость по регионам.

## 2.2 Telegram-направление: telemt + MTProxy

## telemt: заметный рывок в security hardening

Это один из самых “безопасностно насыщенных” апстримов 2026 в вашем наборе.

### Ключевые изменения

#### 1) Hardening FakeTLS/masking pipeline
**Что это даёт:**
- Снижение успешности активного probing-а и классификации “это не настоящий TLS”.
- Более корректная обработка пограничных размеров/типов TLS records (в т.ч. для Telegram-подобного трафика).

**Как это работает внутри:**
- В `tls_stream` сделана строгая валидация record type/version/length:
  - ApplicationData допускается до ciphertext-лимита,
  - Handshake/control records проходят более узкие проверки.
- Это уменьшает риск parser confusion и аварий на “злых” пакетах.

#### 2) Массовое добавление adversarial/security тестов
- По структуре `src/proxy/tests/*` видно, что это уже не “пара unit tests”, а комплексный red-team style набор:
  - probing/shape/timing атаки,
  - race/overflow/timeout сценарии,
  - изоляция shared state,
  - проверки поведения в условиях давления/деградации.

**Практический эффект:**
- Более предсказуемое поведение под нагрузкой и при целевых атаках DPI/сканеров.

#### 3) Управление ресурсами и защитные лимиты
- Появились/усилились механики lease/reservation, eviction и timeout для relay path.
- Добавлен Shadowsocks upstream support.

**Для оператора:**
- Проще удерживать стабильность ноды на всплесках соединений, не убивая latency для легитимных клиентов.

### На что обратить внимание

1. Этот проект быстро эволюционирует: обновления тащат не только фичи, но и изменение security semantics.
2. Перед продом обязательно гонять нагрузочные сценарии с probing/saturation (хотя бы в сокращённом виде).
3. Отдельно проверить hot-reload и graceful teardown в вашей операционной схеме.

## 2.3 TrustTunnel-семейство: endpoint + client + flutter

Эти три проекта тесно связаны, и их изменения лучше рассматривать как единый релизный контур.

### TrustTunnel (endpoint): что важно

#### 1) Глубокое развитие deep-link и генерации конфигов
- Поддержка/синхронизация формата `tt://?`, экспорт новых полей, генерация `client_random_prefix` + автоматическое добавление правил.

**Зачем это нужно:**
- Быстрее и безопаснее онбординг клиента: меньше ручных ошибок, лучше согласованность клиент/сервер.

#### 2) Security fix уровня “сетевой изоляции”
- Исправлена утечка трафика в локальную сеть при запрете private-network (UDP/ICMP/SOCKS5 forwarders, учёт IPv4-mapped IPv6).

**Критично для VPN-сервисов:**
- Это прямой риск route isolation. Если сервис обещает “только через туннель”, эта категория багов бьёт по доверию и compliance.

#### 3) Connection limits per client
- Появились лимиты на число одновременных H2/H3 соединений на клиента + глобальные default-лимиты + override в credentials.

**Эффект:**
- Защита от “шумных” клиентов и от некоторых форм DoS/ресурсного голода.

#### 4) Новый `auth_failure_status_code`
- Настраиваемый код ответа на auth-failure (407/405).

**Зачем это может понадобиться:**
- Тонкая подстройка поведения под фронт/совместимость/маскировочную стратегию на edge.

### TrustTunnelClient: акцент 2026

- Deep-link import стал центральным каналом конфигурирования.
- Добавлены/мигрированы поля (`dns_upstreams`, `name` и т.д.), улучшена совместимость платформ (Windows/iOS-sim/macOS).
- Исправления secure DNS certificate validation (включая self-signed сценарии).
- Добавлен пароль для локального SOCKS5 в secure DNS цепочке.

**Администраторам:**
- Нужно синхронно обновлять endpoint + client + deeplink-схему. Частичные обновления дают ломкие edge-cases.

### TrustTunnelFlutterClient

- Добавлены/починены Always-on-VPN (Android), deep-link decoder/plugin, custom SNI, dual stack, secure DNS validation.
- Отдельно важен фикс про неаутентифицированный локальный SOCKS5 bypass (profile isolation/security boundary).

**Практический вывод:**
- Мобильный клиент в 2026 существенно подтянули по security posture; но нужно контролировать версионную совместимость плагина, native libs и deeplink-ffi.

## 2.4 DPI-обходные утилиты: zapret + dpibreak

## zapret (nfqws/tpws ecosystem)

### Что изменилось по сути

- Фиксы в распознавании/обработке QUIC (включая quicv2 и backport QUICDefragCrypto).
- Исправления стабильности TCP поведения при timestamp fooling.
- Исправление потенциального OOB read.
- Подкрутка STUN detector и рабочие правки blockcheck.

### Почему это важно

- Это проект низкоуровневый: даже “маленький” фикс здесь часто = большое изменение в реальной проходимости в сетях провайдеров.
- OOB/read и неверная классификация QUIC — это не косметика, а источник crash/false positives/непредсказуемых bypass-результатов.

## dpibreak

### Ключевые изменения 2026

#### 1) Расширение управления сегментацией: `--segment-order`
**Что это даёт:**
- Можно задавать порядок/границы отправки сегментов пакета более точно.
- Это мощный инструмент против DPI, который ожидает “нормальный” порядок и цельный ClientHello-профиль.

#### 2) Эволюция fake-autottl и hop inference
- Доработан путь сбора SYN/ACK/TTL (в т.ч. rxring на Linux, отдельные WinDivert handles на Windows).
- Улучшена стабильность и диагностика (логирование hop inference, корректные буферы/закрытие handle).

#### 3) Платформенная устойчивость
- RxRing/iptables/nftables доработки, IPv6 SYN/ACK BPF filter, сервисные фиксы под Windows.

### На что обратить внимание операторам

1. `--segment-order` требует эмпирической настройки по стране/провайдеру/ASN — универсального лучшего профиля нет.
2. fake-autottl — полезен, но может ломать часть маршрутов/сценариев; обязательно A/B тестировать с реальной метрикой успеха соединений.
3. При обновлении следите за изменениями CLI-шорткатов/параметров, чтобы не сломать старые systemd/Task Scheduler юниты.

## 2.5 Hiddify App

По логам 2026 здесь доминируют изменения в прикладной плоскости клиента и профилей:

- обновления WARP/free configs/detour-профилей,
- добавление Naive и новых inbound-сценариев (TPROXY/REDIRECT/DIRECT-порты),
- добавление опции TLS fragmentation packet,
- активная починка кроссплатформенной сборки (Android/AAB, Windows, AppImage, CI).

### Что важно по сути

1. **Функционально:** расширили набор обходных схем “из коробки” (Naive, WARP-варианты, inbound-режимы).
2. **С точки зрения обхода блокировок:** добавление TLS fragmentation option и detour-вариантов даёт больше тактических комбинаций для сложных сетей.
3. **Для админа VPN-сервиса:** Hiddify здесь выступает как “операционный клиентский фронт”; критично контролировать совместимость с backend-конфиг генератором и качество дефолтных профилей.

## 3) Прогноз развития

## Что вероятнее всего увидим дальше

1. **Дальнейший уход от небезопасных “обходных” компромиссов в TLS**
   - Тренд уже начался: pinning/strict verification вместо `allowInsecure`-практик.
   - Ожидаю дальнейшего tightening по конфиг-валидации в Xray/sing-box экосистемах.

2. **Усиление адаптивной маскировки на уровне поведения, а не только сигнатур**
   - По telemt видно направление: shape/timing/adversarial defense.
   - Вероятно, больше runtime-адаптации под профиль сети (dynamic envelopes, policy by feedback).

3. **Рост значения TUN/system-interface пути**
   - Всё больше функций будет крутиться вокруг системного перехвата трафика, pre-match и policy bypass действий.
   - Для VPN-админов это плюс (единая политика), но и новые зоны риска (утечки, несовместимость с local DNS/OS routing).

4. **Более “контрактный” API и стабильность схем deep-link/конфигов**
   - TrustTunnel-семейство в 2026 показало активную нормализацию полей и форматов.
   - Скорее всего, дальше будут строже versioned схемы и автоматические миграторы.

## Практическая стратегия для вашей команды

1. Зафиксировать baseline-профили по странам/провайдерам и регулярно перетестировать после апдейтов (особенно transport + TUN + DNS).
2. Разделить rollout на каналы:
   - security-hotfix канал,
   - feature канал,
   - experimental DPI-bypass канал.
3. Внедрить “обязательный preflight” перед релизом:
   - DNS leak check,
   - local network isolation,
   - fallback transport matrix,
   - deep-link backward compatibility,
   - long-session stability (QUIC + TCP).

## Мини-чеклист «что обновлять в первую очередь»

1. **Xray узлы**, где ещё встречается `allowInsecure`.
2. **TrustTunnel endpoint/client**, если важна строгая изоляция локальных сетей и корректный deep-link import/export.
3. **telemt ноды**, если у вас высокий риск активного probing-а.
4. **dpibreak/zapret профили**, если критична живучесть в сетях с частыми изменениями DPI-правил.
