## Что изменилось в TrustTunnel (серверное/ядро)

Источник: `projects/changelogs/TrustTunnel.log`, PR **#197** (коммит `3e1514f...`, дата `2026-04-27 08:00:44 +0000`).

### 1) Исправлена обработка таймаутов QUIC/H3 на Linux

**Проблема до исправления**
- В мультиплексоре QUIC (компонент, который держит много QUIC-соединений поверх одного UDP-сокета) таймауты восстановления потерь могли не доводиться до повторной отправки пакетов.
- Это влияло на большие H3-передачи: поток мог «зависать» до idle-timeout сервера.

**Что сделали технически**
- Добавили событие `RefreshDeadline` от сокета к мультиплексору, чтобы пересчитывать дедлайн (`quiche::Connection::timeout`) после отправки буферизованных данных.
- В `process_timeouts()` теперь после `on_timeout()` сразу вызывается `flush_pending_data()`, затем пересчитывается новый timeout.
- Логика `closest_deadline` сделана более корректной: теперь выбирается минимальный дедлайн среди активных соединений.

**Почему это важно**
- Для эксплуатации: повышается стабильность HTTP/3-трафика в условиях потерь/джиттера.
- Для R&D по обходу DPI: уменьшается вероятность «ложных обрывов» в длинных сессиях и при агрессивных сетевых условиях.

### 2) H3-тесты перестали быть macOS-only

- Из тестов сняты ограничения `#[cfg(target_os = "macos")]` для H3-сценариев.
- Практический эффект: Linux CI и локальные Linux-стенды теперь ловят регрессии в HTTP/3 раньше.

### 3) Отдельный автоматический bump версии endpoint

## Что изменилось в TrustTunnelClient (клиентские SDK/обвязки)

Источник: `projects/changelogs/TrustTunnelClient.log`, PR **#677** (коммит `8c37a1d...`, дата `2026-04-27 12:19:25 +0000`).

### 1) API упрощён: имя сервера больше не передаётся отдельным аргументом

**Было**
- В Apple API и тестовом приложении часть вызовов ожидала отдельный `serverName` плюс `config`.

**Стало**
- `serverName` берётся из самого конфига (`endpoint.name`).
- Методы обновления/старта конфигурации работают только через `config`, что уменьшает риск расхождения между «именем в аргументе» и «именем в JSON-конфиге».

**Зачем это сделали**
- Убрали дублирование источника истины.
- Снизили вероятность ошибок интеграции в мобильных клиентах и тестовых обвязках (Flutter/Pigeon bridge).

### 2) `dns_upstreams` теперь опционален в Apple-конфиге

- Поле `endpoint.dns_upstreams` сделано optional.
- В `PacketTunnelProvider` добавлен безопасный fallback: если upstream DNS не задан, используется дефолтный набор DNS-серверов.

**Практический эффект**
- Конфиги становятся более совместимыми (меньше обязательных полей).
- Снижается риск падения на этапе парсинга/старта туннеля из-за отсутствующего ключа.

### 3) Автоматический bump версии клиента

Источник: `projects/changelogs/TrustTunnelClient.log`, коммит `787ad43...` (`2026-04-27 12:19:33 +0000`).

## Влияние на пользователей

1. **TrustTunnel (H3/QUIC) должен вести себя стабильнее на Linux** при сложной сети и потерях.
2. **Клиентская конфигурация проще**: меньше параметров в публичном API запуска.
3. **Выше устойчивость к неполным конфигам** за счёт optional `dns_upstreams`.

---

## На что обратить внимание разработчикам

1. Если у вас есть обёртки над TrustTunnelClient API:
   - проверьте, что вы больше **не передаёте `serverName` отдельно**;
   - убедитесь, что в конфиге присутствует `endpoint.name`.
2. Если генерируете конфиги автоматически:
   - валидируйте `endpoint.name` как обязательное поле;
   - `dns_upstreams` можно не задавать, но тогда будет fallback на дефолтный DNS.
3. Для серверной части:
   - обновите regression-набор Linux-тестов H3 и добавьте сценарии с packet loss/задержками.
